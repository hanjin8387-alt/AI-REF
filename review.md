# PROMETHEUS 리뷰 보고서 (비전공자용)

- 기준일: 2026-02-09
- 문서 버전: v2.0 (전면 개편)
- 상세 근거 문서: `review-evidence.md`

## 1. 프로젝트 한 줄 소개
PROMETHEUS는 사진으로 식재료/영수증을 인식하고, 재고를 관리하며, AI 레시피를 추천하는 스마트 키친 앱입니다.

## 2. 이번 리뷰의 결론 5줄 요약
1. 기존 리뷰는 방향은 맞았지만, 보안 위험을 실제보다 낮게 평가한 부분이 있었습니다.
2. 특히 인증 구조와 토큰 관리 방식은 즉시 대응이 필요한 상위 위험입니다.
3. 데이터 정합성 측면에서는 조리 완료 경로의 트랜잭션 부재가 핵심 리스크입니다.
4. 성능과 유지보수성 이슈(N+1, 중복 계산, 큰 화면 컴포넌트)도 중기적으로 누적 비용이 큽니다.
5. 이번 문서는 "문제-영향-조치" 중심으로 재구성했고, 모든 핵심 주장에 코드 근거를 연결했습니다.

## 3. 지금 당장 막아야 할 위험 3개

### 3-1. `SEC-01` 인증 구조 취약점 (보안 축, S0)
공유 앱 토큰과 클라이언트 헤더 기반 `device_id` 신뢰만으로 데이터 접근을 제어하면, 토큰 유출 시 타 사용자 데이터 접근 가능성이 생깁니다.

### 3-2. `DATA-01` 조리 완료 경로의 데이터 불일치 위험 (데이터 축, S1)
재고 차감/삭제와 요리 이력 기록이 분리 실행되어 중간 실패 시 기록 불일치가 발생할 수 있습니다.

### 3-3. `SEC-03` 내부 예외 메시지 외부 노출 (안정성/보안 축, S1)
내부 예외 문자열이 API `detail`로 바로 노출되면, 운영 구조와 내부 정보를 불필요하게 드러낼 수 있습니다.

## 7. 용어사전 (비전공자용)
- 인증(Authentication): "누구인지 확인"하는 절차
- 권한(Authorization): "무엇을 할 수 있는지"를 정하는 규칙
- 토큰(Token): 서비스 출입증 역할을 하는 문자열
- 트랜잭션(Transaction): 여러 작업을 하나로 묶어 "모두 성공 또는 모두 취소"를 보장하는 방법
- N+1 쿼리: 반복문마다 DB를 다시 호출해 느려지는 패턴
- CORS: 어떤 웹사이트가 이 API를 호출할 수 있는지 정하는 브라우저 보안 규칙
- Rate Limiting: 짧은 시간에 들어오는 과도한 요청을 제한하는 장치
- 구조화 로깅: 검색/분석이 쉬운 형식(JSON 등)으로 로그를 남기는 방식
